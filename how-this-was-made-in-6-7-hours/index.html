How this was made (in 6–7 hours)

A walkthrough of like67.com — built in an afternoon by talking to an AI.

This site is a joke. A page for the moment when somebody says like 6… 7. The whole thing — the HTML, the animations, the icons, the share image, the install prompt for your phone — was built in one afternoon by typing into a chat with an AI. I didn't write any code by hand. Even this tutorial and recap was generated without typing any code.

Below is the story of how it actually happened, in the order it happened, broken into seven pieces.

Each piece has two columns. The left column is for someone who is just curious in seeing the pieces of what I did, in plain English. The right column is for someone who wants to learn a little more technical language (jargon) with the goal of being detailed enough to copy the whole thing — the specific tools, the prompts I used word-for-word, the commands I ran. You don't need to read both.

Anything underlined with little dots, like HTML, is an acronym — hover (or tap) to see what it stands for.

How it went

In plain English
To replicate it
Where I was

Nothing existed yet. Just an idea for a stupid webpage about a meme. I wanted a single page that felt like the moment somebody says like 6… 7.

No code, no website, nothing yet. Just an idea and a closed laptop.

What I wanted

One web page that, when you opened it, felt like the meme. Big glowing “6 7” numbers in the middle. Confetti. Color. Noise. Nothing useful. Just chaos pointed at a single moment.

A self-contained index.html — markup, styles, and scripts all in one file. No assembly step before it could run (a “build step”), and no extra library like React to install first (a “framework”). Something I could double-click on my own machine and have the whole site running locally before anything else got involved.

What I already knew (that might be new to you)

There's a pile of background knowledge baked into even the first sentence I typed. I knew that webpages are made of code called HyperText Markup Language (HTML). I knew that the code lives in a file you can save and open. I knew an AI chat could write that code, not just talk about it.

None of those are obvious if no one has ever told you. The whole tutorial below is paced to a reader who is finding out about each of them as they appear.

HTML / CSS / JS basics. That JSFiddle exists as a place to paste-and-run. That LLMs iterate well in conversation — they get better when you say what's wrong. That plain-English prompts work; you don't have to “write code” to ask for code.

What I asked

I told the AI: make me a crazy 6–7 animation webpage. Big numbers, vibrant, more chaos than makes sense. Then I tried what it built, said what was off, and asked for fixes. Over and over. Each round was just a few words.

> Can you do custom html animations?
> I want a 6 7 animation. Something big, vibrant and crazy. Reflecting the meme.
> That's pretty good, can we add anything else to the chaos?
> Make all the hypeText, rare. Let's get rid of the More Chaos button.
What got built

A working webpage. Big numbers in the middle, glowing colors behind them, raining emojis, hype text that flashes once in a while. I could see it run on the screen, try it, and ask for changes. By the end of the conversation, the page was the page.

One HTML file (~200 lines) with embedded <style> and <script>. About ten prompts of iteration. Previewed live in JSFiddle, then the final result was saved as a local file. The first git commit doesn't happen until §2.

Concepts introduced in this section

Click any concept for a definition and an example.
HTML

HyperText Markup Language. The code that tells a browser what's on a webpage — headings, paragraphs, images, buttons. It's just text in a file, with little tags wrapped around the parts.

Example: <h1>Hello</h1> makes the word Hello show up as a big heading.

CSS

Cascading Style Sheets. The code that tells the browser how the page should look — colors, sizes, fonts, animations.

Example: h1 { color: hotpink; } turns every <h1> heading hot pink.

JavaScript

The code that lets a webpage do things — react to clicks, change over time, run animations, respond to the user.

Example: a button that, when clicked, makes confetti fall down the screen.

prompt

What you type to an AI. The clearer and more specific you are, the better the answer. You can iterate — say what's wrong, ask for a change, get a new version.

Example: “Make the numbers bigger and add more emojis.”

iteration loop

The cycle of: ask for something → see what came back → say what's wrong or what you want next → repeat. Most of building with AI is iteration, not one giant prompt.

JSFiddle

A free website where you can paste HTML, CSS, and JS into three boxes and see your webpage run live in a fourth. Useful for testing before you save anything to a file.

It lives at jsfiddle.net.

§2Saving the work

From session 002 — importing the prototype into a repo

How it went

In plain English
To replicate it
Where I was

The page existed only inside a chat window. If the tab closed or the conversation got buried, it could be gone. Nothing was saved anywhere I fully controlled.

One HTML file assembled from code the AI produced — existing only as a chat message. Not saved to disk, not tracked by any history-keeping software, not backed up anywhere.

What I wanted

To save the work somewhere permanent — a place where every change is recorded, I can roll back to any older version, and the full history is visible to anyone who's curious.

A file called index.html saved into a project folder tracked by a program called Git — software that records a full history of every change you make, like an undo button that goes back years. Then uploaded to GitHub, a website that stores that history online so it's backed up and publicly browsable.

What I already knew (that might be new to you)

I knew that saving a file means putting it somewhere specific — not just in a chat window. I'd heard of GitHub as "where developers put their code," without knowing much more than that.

I had used Git and GitHub before on other projects — so I already knew the basic loop: commit (take a snapshot of the current state) and push (send it to GitHub for safekeeping and sharing). Not everyone starting out will have this background, and that's fine — the concepts section below covers what each step actually does.

What I asked

I asked the AI to help me get the prototype documented and saved into a real project. We figured out what was missing and filled it in together.

> Now I'd like to document the conversation I've already had with ChatGPT in order to preview the website before creating the project.

> This session is 002 — we can copy session 000 basically verbatim from the transcripts I will provide you.
What got built

The HTML file was saved as a real file in a real folder. That folder was connected to GitHub. Anyone who visits the GitHub page can now see the code, download it, and watch the full history of every change ever made.

index.html saved at the top of the project folder, with Git recording a snapshot of that moment. The original ChatGPT conversation preserved alongside it as a companion file. Three short commands in the terminal did the work: git add (choose which files to include in the snapshot), git commit (save the snapshot with a short message describing what changed), git push (send it up to GitHub so it's online and backed up).

Concepts introduced in this section

Click any concept for a definition and an example.
Git

A version control system. It watches a folder on your computer and lets you save named snapshots of everything in it. You can go back to any snapshot at any time.

Think of it like "track changes" in a document editor, but for your whole project folder.

repo

Short for "repository." A folder that Git is tracking. When you run git init in a folder, that folder becomes a repo. Everything inside it — files, subfolders — is now part of the record Git keeps.

commit

A saved snapshot of your project at one moment in time. You give each commit a short message describing what changed. The history of all commits is the full record of how the project evolved.

Example: git commit -m "Session 002: import the prototype"

push

Uploading your local commits to GitHub (or another remote). After you commit, the snapshot exists on your machine. git push sends it up so it's online and backed up.

GitHub

A website that hosts Git repos online. It stores the full history, lets others browse or download the code, and is where Cloudflare Pages (the host we use in §3) watches for changes to deploy.

Tools I used

Git for version control · GitHub to host the repo online · Terminal (the command-line app on Mac) to run git commands.

Commands I ran

git add index.html sessions/ — stage the files to include in the next commit
git commit -m "Session 002: import ChatGPT prototype" — save the snapshot with a message
git push — upload to GitHub

How it went

In plain English
To replicate it
Where I was

The code was on GitHub, but nobody could actually visit the website. If you clicked the GitHub link, you'd see code — a bunch of text in a file. I wanted a real URL that just opens the page.

index.html was in the GitHub repo. No host configured, no domain pointed anywhere. The domain like67.com had been registered at Namecheap but was still pointed at nothing.

What I wanted

A static hosting service with: free tier, auto-deploy on every git push, globally distributed (fast for visitors anywhere), and custom domain support. One-time setup; from then on, "deploy" means git push.

What I already knew (that might be new to you)

I knew that websites have to live somewhere — a "host" — and that hosting sometimes costs money. I knew that domain names have to be pointed at the right place somehow, but I didn't know what that process was called.

That "static hosting" means serving plain files (no server-side code or database needed). That a CDN duplicates the site across servers worldwide so every visitor gets a fast copy. That DNS is the system that maps a domain name to a server's address.

What I asked

I asked the AI to help me pick a host and get the site live. We talked through the options and landed on Cloudflare because it's free, fast, and automatically re-publishes the site every time I push a code change.

> We've got a mockup, we've got documentation, and we've got a github repo. So, we should figure out who we're using for hosting, that way we can setup DNS. I've already purchased a domain through namecheap.

> I think that CloudFlare might be the way to go.
What got built

like67.com went live. We also added a license to the code — a statement of what others can do with it (learn from it: yes; copy it for a business: no). From this point on, every time I push a change, the site updates automatically.

Cloudflare Pages linked to the GitHub repo — it watches for new pushes and automatically re-publishes the site when it detects one. A LICENSE file added to the project folder. Cloudflare set as the domain's nameserver (the entry in the internet's address book that says "when someone visits like67.com, send them to Cloudflare") so DNS, SSL encryption, and the host all live in one dashboard.

Concepts introduced in this section

Click any concept for a definition and an example.
hosting

A computer (or network of computers) that keeps your website running 24/7 and sends the files to anyone who visits. Your laptop could technically be a host, but it'd have to stay on and connected forever. Hosting services do that for you.

DNS

Domain Name System — the internet's address book. When you type "like67.com," your browser asks DNS: "what's the address for that?" DNS replies with a number (an IP address) and your browser goes there.

Pointing DNS means telling the address book which server to send visitors to.

CDN

Content Delivery Network. Instead of one server in one city, a CDN keeps copies of your site on servers all over the world. A visitor in Japan gets the copy from a Japanese server. Faster for everyone.

deploy

Publishing a new version of your site so visitors see it. With Cloudflare Pages, deploying is automatic: push your code to GitHub, Cloudflare detects the change and re-publishes within about a minute. "Deploying" in the session logs just means git push.

license

A statement of what others are allowed to do with your work. Without a license, a public repo is technically all-rights-reserved even though the code is visible.

This project uses CC BY-NC 4.0: you can share and learn from it, but not use it commercially, and you must credit the original.

SSL

Secure Sockets Layer — the encryption that puts "https://" in front of URLs. It means the connection between a visitor's browser and the server is encrypted. Cloudflare handles this automatically.

Cloudflare Pages for hosting · Namecheap for domain registration · GitHub as the source the host watches.

§4Looking right on phones

From sessions 004 & 005 — title, mobile layout, share previews, chant fix
View the raw conversation that produced this →
The unedited prompts and replies. This page is the curated version.

How it went

In plain English
To replicate it
Where I was

The site was live — but when I pulled it up on my iPhone, pieces were cut off. The top of the color spiral was clipped. The "SIX SEVEN SIX SEVEN" chant sat below the visible area. When I texted the link, the preview card said "6 7 Meme Chaos Animation," which read as accidentally bad.

Root cause of the clipping: a known iOS Safari bug where the page's height is calculated before the address bar collapses out of the way — so the page thought it was taller than what was actually visible on screen, and the top of the animation got cut off. The page also had no OG meta tags, so iMessage and social apps had nothing to work with and showed a text-only preview with an auto-generated title.

What I wanted

The page should fill the whole phone screen with nothing cut off. The title in iMessage should be intentional. On a narrow phone screen, the chant should feel like it was designed for mobile — not just squished.

A fix for the 100vh / viewport clip. A proper <title> tag and OG meta tags for link previews. A responsive chant layout. A staging branch so changes could be verified on a real phone before going live.

What I already knew (that might be new to you)

I knew that phones have smaller screens and that sometimes things look different on them. I'd noticed that pasting a link into iMessage shows a preview card — and I knew there was a way to control what appears in it.

That "responsive design" means layout adapts by screen size, via CSS media queries. That OG meta tags (og:title, og:description, og:image) control what social apps and iMessage display. That Git branches let you test changes in isolation before merging to production.

What I asked

I described exactly what I was seeing on the phone — what was cut off, what looked wrong — and we fixed each problem one at a time. I also asked how to test on a preview version of the site before the changes went live.

> When sending a link through iPhone, the title of the page is "6 7 meme chaos". I think we need a much better title. Something that doesn't seem accidentally cringe.

> On an iPhone the "six seven six seven" words are cut off below the screen, and the spiral of colors ends prematurely at the top.

> The font for the "Six seven six seven" part seems small on mobile. Ideally I'd also like the top bar to be black, where the phone time is listed.

> For mobile devices, maybe something that says "six seven" above the 6 7 and "six seven" below it as well?
What got built

The page now fills the iPhone screen edge to edge. The iMessage preview says "like 6… 7." On a phone, the chant wraps above and below the big numbers. The status bar at the top of the phone is black. All of this was tested on a real phone via a preview URL before going live.

The page height was switched to a newer CSS value (100dvh — "dynamic viewport height") that recalculates correctly after the address bar collapses. Page title changed to like 6… 7. Full OG meta tag suite added. A theme-color meta tag set the phone's status bar background to black. The chant was split into a desktop version and a mobile version, each shown only at the right screen width. All changes tested on preview.like67-com.pages.dev before merging to main.

Concepts introduced in this section

Click any concept for a definition and an example.
viewport

The visible area of the browser window. On a phone, that's the screen minus any browser chrome (address bar, navigation). 100vh means "100% of the viewport height" — but iOS Safari calculated this wrong for years, which is why pages got clipped at the top.

100dvh (dynamic viewport height) is the fix: it recalculates when the browser chrome collapses.

Open Graph

A set of meta tags that tell iMessage, Twitter, Discord, and other apps what to show when your link is shared. og:title sets the headline. og:description sets the preview text. og:image sets the thumbnail.

Without them, apps guess from the page content — and they usually guess badly.

meta tag

A line in the <head> of an HTML page that gives browsers and crawlers information about the page — without displaying anything visible on screen.

Examples: page title, description, share image, theme color, viewport settings.

responsive design

Making a page adapt its layout based on screen size. The main tool is a CSS media query: code that says "only apply these styles if the screen is narrower than X pixels."

On this site: at full width, the chant is a single row. On a phone, it splits above and below the big numbers.

branch

A separate copy of the code you can work on without affecting the live site. When the changes look right, you merge the branch back into main.

Cloudflare Pages automatically builds every branch and gives it its own preview URL, so you can test on a real phone before shipping.

Tools I used

Text editor (AI-written changes applied to index.html) · iPhone for real-device testing · preview.like67-com.pages.dev as the staging URL (auto-created by Cloudflare for every branch).

Commands I ran

git checkout -b preview — create and switch to a testing branch (run once)
git push -u origin preview — push the branch so Cloudflare builds a preview URL
git add index.html && git commit -m "Session 004: title, OG tags, iOS viewport fix"
git push — updates the preview URL automatically
After confirming on phone: git checkout main && git merge preview && git push

§5Save it as an app

From session 007 — fullscreen hints and PWA install

The unedited prompts and replies. This page is the curated version.

How it went

In plain English
To replicate it
Where I was

The site worked great in a browser. But visiting it on your phone meant typing the URL or digging up a bookmark — it lived on the web, not on your home screen. And there was no hint anywhere telling people they could go fullscreen on a desktop.

No PWA manifest, no app icons, no Apple-specific meta tags. The double-click fullscreen toggle already existed in the code but was invisible — no label pointed at it. Same for the home screen install gesture on mobile.

What I wanted

Two things: a small hint on desktop telling people to double-click for fullscreen, and a way for people on phones to save the site to their home screen so it opens like a real app — no browser bar, no address bar, just the page filling the whole screen.

A PWA manifest (manifest.json) — a small config file browsers read to know how to treat the site as an installable app. Icons for the home screen. Apple-specific meta tags (iOS doesn't fully follow the standard PWA spec). Hints that appear only on the right kind of device.

What I already knew (that might be new to you)

I'd noticed that some sites — when you save them to your home screen on iOS — open like apps, without the Safari address bar. I knew this was possible but not what it took to set up.

That PWA stands for Progressive Web App — a website browsers can treat like an installed app. That manifest.json is the file that describes the app to the browser. That iOS requires a separate set of Apple meta tags because Safari has its own rules that predate the PWA standard.

What I asked

I described what I wanted and asked the AI to wire it up — including making the hints smart enough to show only on the right kind of device (mouse vs. touchscreen).

> Let's add a small subtle message in the bottom-left on desktop computers to let users know that double-clicking will open full screen. This should only display when on a desktop browser window.

> There are other things we could do for mobile. Like adding a mention about saving a shortcut to the desktop. We can make this a Single Page App this session.
What got built

The site can now be saved to your phone's home screen and opens like an app — no address bar, no browser tabs, just the animation. When you first visit on a phone, a small hint fades in at the bottom telling you how to install it. On desktop, a different hint tells you to double-click for fullscreen. Both quietly fade out after a few seconds.

manifest.json created with display mode standalone, black theme, and both icon sizes. Two icon files generated: assets/icon-192.png and assets/icon-512.png — deep purple background with hot-pink/cyan "6 7." Apple meta tags added to <head>. Two hint labels in the page, each hidden by default and revealed only on the right device type via CSS media queries (hover: hover for mouse, hover: none for touch). JS checks if the app is already installed and suppresses the mobile hint if so.

Concepts introduced in this section

Click any concept for a definition and an example.
PWA

Progressive Web App. A website that browsers can treat like an installed app — it gets a home screen icon, opens in a standalone window without a browser bar, and can work offline. The user doesn't install it from an app store; they save it directly from the browser.

manifest.json

A small config file that tells the browser how to treat the site as an app. It specifies the app's name, icons, background color, and how it should display when launched. The browser reads it when someone tries to install the site to their home screen.

standalone mode

When a PWA opens without any browser UI around it — no address bar, no tabs, no browser back button. Looks just like a native app. Set in manifest.json as "display": "standalone".

media query (pointer type)

A media query is CSS code that only applies under certain conditions. Here we used pointer type to detect the input device: (hover: hover) means a mouse is present; (hover: none) means it's a touchscreen. That's how the right hint appears on the right device.

Tools I used

ImageMagick (command-line image tool, used by the AI to generate the icon PNGs) · manifest.json written by the AI, saved directly to the repo · iPhone for testing the home screen install prompt.

Commands I ran

git add -A — stage all new files (icons, manifest, updated HTML)
git commit -m "Session 007: fullscreen hints, PWA manifest, home-screen install"
git push

Concepts introduced in this section

Click any concept for a definition and an example.
analytics

Data about who visits your site and what they do. At a basic level: how many people came today, where did they come from, which pages did they look at? More invasive tools track individuals across the web. Cloudflare Web Analytics only gives you aggregates — totals, not profiles.

cookies

Small files a website stores in your browser to remember things about you. Often used for login sessions (useful) or cross-site tracking (the kind people find creepy). Cloudflare Web Analytics works without any cookies at all.

beacon script

A tiny piece of JavaScript that quietly signals a server when a page loads — that signal is what analytics tools use to count a visit. Cloudflare's beacon is auto-injected by their platform, so you turn it on in the dashboard and it appears on every page without touching the repo.

COPPA

Children's Online Privacy Protection Act — a US law restricting what data can be collected from users under 13. Since this site could easily be visited by kids, using a tracker that collects no personal data keeps it compliant.

How it went

In plain English
To replicate it
Where I was

When you pasted the link into iMessage or Discord, the preview card showed plain text — nothing that captured the visual chaos of the actual site. It looked like any ordinary link.

The OG title and description tags from §4 were in place, but og:image was missing. Without it, social platforms fall back to a text-only card. No image for their scrapers to find.

What I wanted

A thumbnail that captures the chaos — so when someone shares the link, the preview card makes the recipient immediately understand what they're about to click. The image should feel like the site.

A 1200×630 pixel PNG (the standard OG image size). Generated by a script the AI wrote — so if I want to tweak it, I change a line of code and re-run it, same as editing the HTML.

What I already knew (that might be new to you)

I'd seen those preview cards in iMessage and Discord a thousand times. I knew there was a way to control what they show — we'd already set up the title and description in §4. The image was the missing piece.

That og:image is the specific tag that controls the thumbnail. That 1200×630 is the standard size most platforms expect. That Python's Pillow library can generate images from code — which means the AI can write the image generator the same way it writes anything else, and iteration is as fast as any other change.

What got built

A real share image — 1200×630 pixels of the same chaos as the page: dark glowing background, big "6 ⚡ 7" in the centre, the site name at the top, and a "like67.com" button at the bottom. Paste the link anywhere now and that's what shows up.

A Python script (written by the AI, using the Pillow image library) generates the PNG: radial gradient background, coloured glow blobs, scanlines, glowing number glyphs, lightning bolt, headline, tagline, and pill CTA. Saved to assets/og-image.png. Wired into index.html with og:image and twitter:image tags; twitter:card upgraded to summary_large_image for the big-preview format on X/Twitter and Discord. Concepts introduced in this section

Click any concept for a definition and an example.
OG image

The thumbnail that appears when a link is shared on iMessage, Discord, Twitter, Slack, and similar apps. Set by <meta property="og:image"> in the page's <head>. Standard size: 1200×630 pixels.

Without it, platforms either show a blank card or guess an image from the page — which usually looks bad.

Python script

A small program written in the Python language that runs on your computer and does a specific job — in this case, generate an image file from scratch. The AI wrote the script; running it was one command in the terminal. Generating images with code means iteration is instant: change a value, re-run, see the result.

Pillow

A Python library for creating and editing images. A "library" is a collection of pre-written code you can use in your own scripts. Pillow handles drawing shapes, text, gradients, and layering — everything needed to build the OG image programmatically without opening a design tool.

summary_large_image

A Twitter/X card format that shows a big image preview instead of a small thumbnail beside the text. Discord and some other platforms also respect this value. Set via <meta name="twitter:card" content="summary_large_image">.
